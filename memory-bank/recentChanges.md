- **Waste: offline resilience hardening + 14-day home window (2026-07-02)**:
  - **Persistent media queue (single owner)**: queued photos/signatures are now copied into `<appDocsDir>/waste_media_queue/` at queue time (`SyncService.persistWasteMediaForQueue`, deterministic dest name so re-queueing the same capture maps to the same file) — previously the queue held bare paths into the OS-clearable cache, silently losing media on cache clears. The in-memory session queues in `WasteService` (`_sessionOfflinePhotoQueue`/`_sessionOfflineSignatureQueue`) were **deleted** — the central Hive queue is the single owner; the dual-queue design could double-upload (Hive processor didn't delete the file; session pass then re-uploaded under a new UUID). `processOfflineWasteQueue()` now only drains the central queue + heals OFFLINE-* numbers. `migrateQueuedWasteMediaToPersistentDir()` (run from `SyncService.init()`) heals pre-update queue entries.
  - **Loss surfacing**: a queued media file missing at replay writes a `waste_audit` entry (`action: media_lost`, media_type/load_id/item_id/queued_at) instead of vanishing silently; `WasteQueuedScreen` shows a red "could not be recovered" section via new `WasteService.getRecentLostMediaAudit()` (no orderBy — client-sorted, avoids a composite index).
  - **Idempotent replay**: storage filenames in both queue processors are now derived from the queue item id (`{item.id}.jpg` / `signature_{item.id}.png`) so a retry after an ambiguous failure re-uploads the same object instead of minting a duplicate URL for `arrayUnion`. Photo/signature entries are deduped by `localPath` in `_dedupeWasteQueue` (now also run per `_processQueue`, not just init).
  - **photo_count recompute**: replay recomputes a load's `photo_count` from truth (`load_photos` + non-deleted item `photos`) after late photo uploads and after replaying any queued `waste_loads` write carrying `photo_count`/`load_photos` — a queued absolute count is stale by replay time and used to clobber late uploads (Pulse displays photo_count in list/CSV/detail). `finishLoading` also gained the missing online `photo_count` increment.
  - **Timestamp restoration**: queued `waste_loads`/`waste_items` payloads replay through new `SyncService.restoreWasteLoadTimestamps`/`restoreWasteItemTimestamps` (ISO string → `Timestamp.fromDate` for date_time/scheduled_for/completed_at/cost_reviewed_at/weighbridge_received_at/pending_cost_review_at/pending_weighbridge_at/weighbridge_ticket_waived_at/createdAt; updatedAt → serverTimestamp; unparseable strings pass through). Previously offline-created loads landed with **string** dates and fell out of every Pulse Timestamp range query (reports, board metrics) — same bug class fixed earlier for job_cards/waste_stock/fleet. Backfill for existing docs: extended `repairWasteLoads` CF (monorepo).
  - **IBC pool ops online-only guard**: `splitPoolStock`/`removeDamagedIbcUnits` (+ `InkService.transferIbc`) now pre-check connectivity (clear `StateError` message) and run their transactions under a 15 s timeout — they used to hang indefinitely on dead Wi-Fi (Firestore transactions can't run offline and mustn't be queued blind against shared pool state).
  - **Queue error visibility**: fleet's per-item `lastError` map generalized to waste (`_queueLastErrors`); `getQueuedWasteDetails` exposes `lastError`; `WasteQueuedScreen` rows show it ("will retry automatically").
  - **14-day home window (all roles, user-confirmed)**: `watchActiveLoads`/`watchScheduledLoads`/`watchRecentCompleted` filter to the last 14 days (`WasteService.homeListWindow`) — scheduled loads use one `scheduled_for >=` range that covers future-scheduled too.
