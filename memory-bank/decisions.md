# Active Decisions and Considerations

Open architectural/design decisions guiding ongoing work.

- **Employee Display Format**: Using `displayName` (name + clockNo + position) + department for clean, non-redundant UI
- **Filtering Logic**: Department filter overrides mech/elec filter for precise assignment control
- **Real-time Updates**: StreamBuilder for live employee list updates in assign dialog
- **Super-Manager Pattern**: Users with `department == 'general'` treated as super-managers with full oversight - no dept filters by default, access to all filters like dept managers, view all jobs on login
- **Off-site write locks deferred (2026-06-28)**: Server-side `isOnSite` write enforcement for floor roles is intentionally deferred until permission health (`DeviceHealthService`, `employees.permissions` merge, admin On Site indicators) is validated in production. Targeted admin broadcast supports permission-fix outreach in the interim.
- **Waste offline media: single-owner Hive queue + recompute-over-increment (2026-07-02, user-confirmed design)**: (1) The central SyncService Hive queue is the *single owner* of queued photos/signatures — WasteService's in-memory session queues were deleted, not kept as counters, because the dual-queue design double-uploaded (the Hive processor left the file on disk; the session pass then re-uploaded it under a fresh UUID → duplicate URL arrayUnioned). (2) Queued media is copied to `waste_media_queue/` under the app documents dir with a *deterministic* destination name (sanitised source path) so re-queueing the same capture maps to the same file and the new localPath-based dedupe collapses duplicates; Storage object names derive from the queue item id for retry idempotency. (3) `photo_count` is *recomputed from truth* on replay, never incremented — the queued absolute count replays *after* the photo entries it undercounts (Hive insertion order), so an increment would be clobbered; increments also aren't retry-safe. (4) Permanently lost media (file gone before sync) is surfaced via `waste_audit` `media_lost` entries + a red WasteQueuedScreen section — chosen over a local Hive tombstone so Pulse sees the same record. (5) IBC pool transactions (`splitPoolStock`/`removeDamagedIbcUnits`/`transferIbc`) stay **online-only** (connectivity pre-check + 15 s timeout + clear StateError copy), explicitly NOT queued offline — shared read-modify-write pool state must not be replayed blind; a timed-out transaction may still commit, so the non-idempotent ops' copy says "check the stock list before retrying". (6) Mobile waste home lists window to the last 14 days (+ future-scheduled) for *all* roles, server-side — managers use Reports/Pulse for history; also cuts Firestore reads.
- **Waste Scheduled-vs-Create-from-scratch convergence rules (2026-06-30, user-confirmed via 2 rounds of clarifying questions)**: (1) `paper_document_ref` required everywhere it's actually captured at collection time — Schedule screen's own field stays optional (pre-fill hint only). (2) On-site stock pick converges to "any waste user" (not admin/manager-only) on all three relevant screens. (3) Manager's `selected_waste_types` is *enforced* at Begin Collection, not just informational — admin-only override widens back to the full contractor list, usage recorded to `waste_audit` for manager review. (4) Signature/photo requirements are settings-driven (`waste_settings`) on both paths — neither hardcoded mandatory nor hardcoded optional. (5) `createWasteLoad` CF retry/duplicate risk gets the full fix (client-generated `client_ref` dedup check server-side), not just defensive logging — chosen because the risk is real for both creation paths, not just the rewritten one. (6) `trailerReg` modeled as nullable `String?` (genuinely optional everywhere) rather than `vehicleReg`'s non-null-with-`''`-default pattern, since unlike driver/vehicle it's never required on any path.
- **Ink IBC consolidated waste-pool design (2026-06-30, user-confirmed + independently validated by a Plan agent)**: (1) Partial selection allowed when linking on-site IBC quantity to a load (not all-or-nothing) — confirmed with the explicit addition that a *damaged/scrapped* removal must capture a reason, distinct from a plain partial take. (2) The consolidated pool tracks individual physical `ink_ibcs` numbers (`linked_ibc_numbers[]`), not a bare count — recommended option, preserves traceability for damaged-removal and reporting. (3) Damaged-removal lives only at Begin Collection (no standalone "mark on-site stock damaged" screen) — matches the user's own framing ("remove these... from the load"). Technical follow-ons found during Plan-agent validation, folded in rather than left as risks: a transactional *query* for "does an open pool exist" doesn't serialize against a racing transaction doing the same query (two concurrent consumes could both create a pool) — fixed with a deterministic pointer doc (`waste_stock_pool_pointers/ibc_bins`), the same pattern already used for `waste_counters`. Partial/full takes always split into a new doc (never reuse-and-flip the pool doc directly) specifically to avoid needing to clear/repoint the pointer on a "take everything" pick — the pool doc just keeps existing, possibly at quantity 0, as the stable ongoing accumulator. `disposed` status is not reused bare for damaged-removal (it already means "voided/manually disposed" elsewhere) — a separate `damaged` boolean discriminates the two. `InkService.voidIbcTransfer()`'s stock lookup (`assertIbcStockVoidable`/`disposeIbcStockOnVoid`) had to be redesigned in the same pass — it keyed off the now-obsolete deterministic per-IBC doc id and would have silently no-op'd otherwise. Pulse's web stock picker (`WasteStockPicker.tsx`) is explicitly out of scope (mobile-only for now, user-confirmed) — it has the same one-doc-per-IBC assumption but partial-take UI wasn't extended there.
