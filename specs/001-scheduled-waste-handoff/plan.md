# Implementation Plan: Scheduled Waste Load Handoff

**Branch**: `001-scheduled-waste-handoff` | **Date**: 2026-06-01 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/001-scheduled-waste-handoff/spec.md`

---

## Summary

Add a two-phase waste load lifecycle: Security Managers schedule incoming loads (contractor, type, date) before the truck arrives; Security Guards complete the collection at the gate (driver, reg, items, photos, signature). A new `pending_weighbridge` status bridges guard submission to manager weighbridge entry. The existing single-session guard-creates flow is preserved alongside the new flow.

---

## Implementation Phases

### Phase A — Model & Service (Foundation)

**A1. Extend WasteLoadStatus enum** (`lib/models/waste_load.dart`)

Add after `completed`:
```dart
scheduled('scheduled'),
pendingWeighbridge('pending_weighbridge'),
cancelled('cancelled');
```

Update `fromString` / factory constructor to handle new values with fallback to `draft`.

**A2. Extend WasteLoad model** (`lib/models/waste_load.dart`)

Add 6 nullable fields (see `data-model.md`). Update both `fromFirestore` and `toFirestore`. Existing fields untouched.

**A3. Add `createScheduledLoad()`** (`lib/services/waste_service.dart`)

Simple `addDoc` call (no Cloud Function needed — no load number required for scheduling). Sets `status: 'scheduled'`, `scheduled_for`, `scheduled_by`, `scheduled_by_name`, `scheduled_notes`, `created_by`, `is_deleted: false`.

**A4. Add `watchScheduledLoads()`** (`lib/services/waste_service.dart`)

Stream query: `status == 'scheduled'`, ordered by `scheduled_for` ascending, limit 50.

**A5. Add `cancelScheduledLoad()`** (`lib/services/waste_service.dart`)

Firestore transaction: read → assert `scheduled` → write `cancelled`. Throws if already started.

**A6. Add `submitCollection()`** (`lib/services/waste_service.dart`)

Transaction for status transition (`scheduled` → `pending_weighbridge`), then photo uploads and item batch-writes reusing existing helpers. Queues each operation via `SyncService`.

**A7. Update `watchPendingWeighbridge()`** (`lib/services/waste_service.dart`)

Change query to `status == 'pending_weighbridge'`, order by `pending_weighbridge_at` descending. Remove 3-day threshold and client-side weighbridge-number filter.

**A8. Add Firestore indexes** (`firestore.indexes.json`)

Two composite indexes (see `data-model.md`). Deploy with `firebase deploy --only firestore`.

### Phase B — Manager Schedule Screen (New Screen)

**B1. `WasteScheduleLoadScreen`** (`lib/screens/waste_schedule_load_screen.dart`)

`ConsumerStatefulWidget`. Fields:
- Contractor dropdown (uses existing `watchContractors()`)
- Main waste type grid (uses existing `watchWasteTypes()`)
- Date picker for expected date (`scheduledFor`)
- Notes text field (optional)

On save: calls `WasteService.createScheduledLoad()`. Shows success snackbar, pops screen.

Validation: contractor + waste type + date required. Save button disabled until valid.

**B2. Wire entry point in `waste_home_screen.dart`**

For Security Managers and Admins: replace single FAB with a `SpeedDial` (or `FloatingActionButton.extended` menu) offering:
- "Schedule Incoming" → `WasteScheduleLoadScreen`
- "New Load" → existing `WasteCreateLoadScreen` (legacy path)

For Security Guards: FAB unchanged ("New Load" only).

---

### Phase C — Guard "Incoming" Section (Home Screen Update)

**C1. Add `watchScheduledLoads()` stream to `waste_home_screen.dart`**

Add a second `StreamBuilder` widget at the top of the scroll content. Only renders when the stream returns a non-empty list.

**C2. Incoming load card widget**

Each card shows:
- Contractor name
- Waste type (with type icon)
- Expected date + time
- Manager notes (if any)
- "Begin Collection →" button (primary orange CTA)

On tap: navigates to `WasteBeginCollectionScreen(load: scheduledLoad)`.

---

### Phase D — Guard Begin Collection Screen (New Screen)

**D1. `WasteBeginCollectionScreen`** (`lib/screens/waste_begin_collection_screen.dart`)

`ConsumerStatefulWidget`. Receives `WasteLoad` (in `scheduled` status).

**Read-only header**: contractor name, waste type, scheduled date, manager notes.

**Guard fills**:
- Driver name (required text field)
- Vehicle registration (required text field)

**Waste items section** (reuse existing `WasteItemEntryDialog` pattern):
- Add item button → subtype, weight, quantity, photos
- At least 1 item with 1 photo required before submit

**Signature**:
- "Capture Signature" button → navigates to existing `WasteSignatureScreen`
- Signature preview shown once captured

**Submit button**:
- Disabled until: driver name, reg, ≥1 item with photo, signature
- On tap: calls `WasteService.submitCollection()`
- Shows loading state, success → pops to home, error → shows retry option

**E1. No UI changes needed** — `waste_pending_weighbridge_screen.dart` already uses `watchPendingWeighbridge()` stream. After A7, it will automatically show loads in `pending_weighbridge` status.

**E2. Update display labels** — "Pending Weighbridge" cards should show the guard's `collectedBy` name and `pendingWeighbridgeAt` timestamp in addition to existing fields.

### Phase F — Cancel Scheduled Load (Manager)

**F1. Add cancel action to Incoming load card** (Phase C widget)
